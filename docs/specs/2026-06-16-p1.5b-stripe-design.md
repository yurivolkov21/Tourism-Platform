# P1.5b — Stripe checkout + webhook + refund · design spec

**Date:** 2026-06-16 · **Branch:** `feat/p1.5b-stripe` · **Phase:** P1.5b
**Refs:** [ADR-0006](../decisions/0006-multi-gateway-momo.md) (Stripe + PayPal) ·
[ADR-0008](../decisions/0008-security-integrity-hardening.md) (integrity) · donor
`modules/payments`. Depends on P1.5a (bookings core; `BookingsService` exported).

## Goal

Wire **Stripe** onto the P1.5a booking lifecycle: mint a hosted Checkout session
for a PENDING booking, confirm payment via a signature-verified webhook (idempotent,
oversell-safe **atomic seat reservation**), and support admin refunds. Provider-
neutral at the data layer (`PaymentProvider.STRIPE`, `providerSessionId` /
`providerPaymentId`, `PaymentEvent.provider`). PayPal reuses this shape in P1.5c.

## Decisions baked in (confirmed with owner)

- **Checkout = separate endpoint** `POST /bookings/:code/checkout` (not extending
  `POST /bookings`). Keeps the P1.5a create contract; supports checkout retry; clean
  for multi-gateway (same booking → Stripe or PayPal session).
- **Email deferred** to the pg-boss outbox phase (ADR-0007, P1.x) — webhook must not
  block on Resend. P1.5b only logs the PAID/REFUNDED transitions.

## Seat-reservation — pooler-safe atomic claim (key adaptation)

The donor used an interactive `$transaction` + `SELECT … FOR UPDATE` on the departure.
Our **Supabase transaction-mode pooler** (port 6543, `connection_limit=1`) assigns a
backend connection **per statement**, which breaks interactive `FOR UPDATE` locks.
Instead, reserve seats in a **single data-modifying CTE** (one round trip, atomic in
Postgres):

```sql
WITH booking_check AS (
  SELECT id, departure_id, (num_adults + num_children) AS seats
  FROM bookings WHERE id = $1 AND status = 'PENDING'
),
seat_claim AS (
  UPDATE tour_departures d SET seats_booked = d.seats_booked + bc.seats
  FROM booking_check bc
  WHERE d.id = bc.departure_id AND d.seats_booked + bc.seats <= d.seats_total
  RETURNING bc.id AS booking_id
)
UPDATE bookings SET status = 'PAID', paid_at = now(), provider_payment_id = $2
WHERE id = (SELECT booking_id FROM seat_claim)
RETURNING id;
```

- Returns 1 row → **paid** (seats claimed + booking flipped, atomically).
- Returns 0 rows + booking still PENDING → **overbooked** (race lost) → auto-refund.
- Returns 0 rows + booking not PENDING → **already processed** (idempotent no-op).

Idempotent on Stripe retries (gated on `status='PENDING'`); oversell-proof (gated on
`seats_booked + N <= seats_total`). No interactive transaction.

## In scope

### Checkout (`POST /bookings/:code/checkout`, JWT, owner-only)
- Load owner's booking (else 404); must be `PENDING` (409 `BOOKING_NOT_PENDING`).
- Provider must be `STRIPE` (else 400 `PROVIDER_NOT_AVAILABLE` until P1.5c).
- Mint hosted Checkout session (`mode=payment`, `customer_email`, single line item
  `unit_amount`×`quantity`, `metadata.bookingId/Code`, `expires_at = now+30m`,
  success/cancel URLs from `FRONTEND_URL`). `unit_amount` via `toStripeMinorUnits`
  (zero-decimal currency aware). Persist `providerSessionId`; return `{ checkoutUrl,
  bookingCode, status }`. On Stripe error, surface 502 `CHECKOUT_FAILED` (booking
  stays PENDING — retryable).

### Webhook (`POST /payments/stripe/webhook`, `@Public` + `@SkipTransform`)
- Raw body via `express.raw({ type: 'application/json' })` mounted on this exact path
  in `main.ts` **before** the global JSON parser. Verify `Stripe-Signature` →
  `constructEvent`; bad/missing → 400 `STRIPE_WEBHOOK_INVALID` (Stripe retries).
- **Idempotency:** insert `PaymentEvent { provider: STRIPE, eventId, type, payload }`
  (`@@unique([provider, eventId])`). Duplicate with `processedAt` set → skip; with
  `processedAt` null → a prior attempt crashed, re-run (booking-level idempotency makes
  it safe). Set `processedAt` when done.
- Dispatch: `checkout.session.completed` → atomic claim (PAID / overbooked-refund);
  `checkout.session.expired` → PENDING→CANCELLED (no seats held); else → log + 200.
- Always 200 on a verified event (even unhandled) so Stripe doesn't retry-storm.

### Admin refund (`POST /admin/bookings/:code/refund`, `@Roles(ADMIN)`)
- Booking must be `PAID` with a `providerPaymentId` (else 400 `BOOKING_NOT_REFUNDABLE`).
- Stripe refund **first** (outside any DB write); converge on `charge_already_refunded`.
- Then atomic: decrement `seatsBooked` + booking→`REFUNDED` (`cancelledAt`,
  `refundReason`, `refundedById` = admin's user id), gated on `status='PAID'`
  (idempotent). On Stripe failure → 400 `REFUND_FAILED` (booking stays PAID for an
  operator to retry).

## Out of scope

- **PayPal** checkout + webhook → P1.5c (reuses this provider-neutral shape).
- **Emails** (confirmation/refund) → pg-boss outbox (ADR-0007, P1.x).
- Abandoned-cart sweep beyond Stripe's own `session.expired` → P1.x.

## Trust / integrity

- Webhook is the only public unauthenticated mutation — authenticated by Stripe's
  signature over the **raw** bytes, never JWT. `@SkipTransform` (Stripe wants a plain
  ack). Checkout + refund are JWT-guarded (owner / ADMIN).
- Secrets from config (`stripe.secretKey`, `stripe.webhookSecret`) — never logged.
- Money: `Decimal` → minor units via `toStripeMinorUnits` (zero-decimal set: JPY/VND/…).

## Verification

- **Unit** — `payments.service.spec.ts`: bad signature → 400; duplicate processed
  event → skip (no side effects); completed → claim path returns paid; overbooked →
  refund + REFUNDED; expired → CANCELLED; unhandled type → 200 no-op. Seat-claim SQL
  invoked via mocked `$queryRaw`. `bookings.service.spec.ts` (extended): startCheckout
  (404/409/PROVIDER_NOT_AVAILABLE, persists session id, returns url via mocked Stripe);
  refundByAdmin (state guards, Stripe-first, already-refunded converge, REFUNDED).
- **Gate** (api scope) green.
- **Smoke:** build → kill stale :3000 → boot. `POST /payments/stripe/webhook` (no
  signature) → 400 `STRIPE_WEBHOOK_INVALID`; `POST /bookings/X/checkout` (no token) →
  401; `POST /admin/bookings/X/refund` (no token) → 401.
