# P1.5c — PayPal (Orders v2) checkout + capture + webhook + refund · design spec

**Date:** 2026-06-17 · **Branch:** `feat/p1.5c-paypal` · **Phase:** P1.5c
**Refs:** [ADR-0006](../02-decisions/0006-multi-gateway-momo.md) (Stripe + PayPal) ·
[ADR-0008](../02-decisions/0008-security-integrity-hardening.md) · P1.5b Stripe (same
provider-neutral shape). SDK: **`@paypal/paypal-server-sdk@^2.4.0`** (GA; the old
`@paypal/checkout-server-sdk` is deprecated).

## Goal

Add **PayPal** as the second rail on the existing booking flow, reusing the
provider-neutral data layer (`PaymentProvider.PAYPAL`, `providerSessionId` =
PayPal **order id**, `providerPaymentId` = **capture id**, `PaymentEvent.provider`)
and the **same atomic seat-claim CTE** + refund machinery from P1.5b. Sandbox creds
verified (OAuth token OK).

## Capture flow (confirmed with owner) — capture-on-return + webhook backstop

PayPal Orders v2 requires an explicit **capture** after the buyer approves:

1. **Checkout** (`POST /bookings/:code/checkout`, provider PAYPAL) → create an
   Order (`intent: CAPTURE`, amount = `booking.totalAmount`/currency, `custom_id =
   bookingId`), persist `providerSessionId = orderId`, return `checkoutUrl` =
   the PayPal **approve** link.
2. Buyer approves on PayPal → redirected to `FRONTEND_URL/checkout/success`.
3. **Capture** (`POST /bookings/:code/capture`, JWT owner) → server captures the
   order → atomic seat-claim + PAID (`providerPaymentId = captureId`). Overbook
   race → auto-refund the capture + REFUNDED.
4. **Webhook backstop** (`PAYMENT.CAPTURE.COMPLETED`) → same atomic seat-claim
   (idempotent no-op if already PAID) so a buyer who closes the tab after paying
   still gets confirmed.

## Shared seat-claim (refactor)

Extract the P1.5b atomic seat-claim CTE into a provider-agnostic
`PaymentsService.claimSeatsForPaid(bookingId, providerPaymentId)` returning
`'paid' | 'overbooked' | 'already_processed'`. The Stripe webhook, the PayPal
capture endpoint, and the PayPal webhook all call it; the **overbook auto-refund is
provider-specific** (handled by each caller). `BookingsService` injects
`PaymentsService` (BookingsModule already imports PaymentsModule → no cycle).

## In scope

### PayPalService (`modules/payments/paypal.service.ts`)

Thin SDK wrapper returning small domain shapes (mirrors `StripeService`):

- `createOrder({ bookingId, bookingCode, currency, amount, returnUrl, cancelUrl })`
  → `{ orderId, approveUrl }`.
- `captureOrder(orderId)` → `{ captureId, status }`; treats `ORDER_ALREADY_CAPTURED`
  as success (idempotent — read the existing capture).
- `refundCapture(captureId)` → `{ id, status }`; converges on already-refunded.
- `verifyWebhookSignature(headers, event)` → boolean via REST
  `POST /v1/notifications/verify-webhook-signature` with `PAYPAL_WEBHOOK_ID` +
  transmission headers (no raw body needed — PayPal verifies the parsed event).
- Internal OAuth token fetch + cache (client_credentials) for the REST verify call.

### Checkout dispatch (extend `BookingsService.startCheckout`)

Dispatch on `booking.paymentProvider`: STRIPE → existing session; PAYPAL →
`createOrder`. Drop the `PROVIDER_NOT_AVAILABLE` 400.

### Capture (`POST /bookings/:code/capture`, JWT owner-only)

`capturePayPal(code, caller)`: owner-load (404), provider PAYPAL (else 400), must
have an order id; if already PAID → return it (idempotent). Capture →
`claimSeatsForPaid`; overbook → `refundCapture` + REFUNDED.

### Webhook (`POST /payments/paypal/webhook`, `@Public` + `@SkipTransform`)

Normal JSON body (no `express.raw`). Verify signature (FAILURE → 400). Idempotency
via `PaymentEvent(PAYPAL, eventId=event.id)`. Dispatch:

- `PAYMENT.CAPTURE.COMPLETED` → resolve booking via `custom_id` + capture id →
  `claimSeatsForPaid` (backstop).
- `PAYMENT.CAPTURE.REFUNDED` / `...DENIED` → log/reconcile (booking already handled
  by admin refund or capture failure). Others → 200 no-op.

### Refund dispatch (extend `BookingsService.refundByAdmin`)

Dispatch on provider: STRIPE → `stripe.createRefund`; PAYPAL →
`paypal.refundCapture`. Same atomic release-seats + REFUNDED CTE; provider-specific
already-refunded convergence.

## Out of scope

- Emails + abandoned-cart sweep → pg-boss outbox (ADR-0007, P1.x).
- Webhook ID acquisition (needs an ngrok URL) — wired during the smoke step; the
  capture-on-return path is testable without it.

## Trust / integrity

- Webhook authenticated by PayPal signature verification (`PAYPAL_WEBHOOK_ID`), not
  JWT; `@Public` + `@SkipTransform`. Checkout/capture/refund JWT-guarded.
- Secrets from config (`paypal.clientId/clientSecret/mode/webhookId`) — never logged.
- Money: `Decimal` → PayPal amount string with the currency's decimal places.

## Verification

- **Unit:** `paypal.service` money/amount formatting; `BookingsService.capturePayPal`
  (404/provider/idempotent-PAID/overbook-refund); `refundByAdmin` PayPal branch;
  `PaymentsService` PayPal webhook (bad-sig 400, dup-skip, capture-completed →
  claim, unhandled → noop); `claimSeatsForPaid` shared (paid/overbooked/already).
- **Gate** (api scope) green.
- **Smoke:** build → boot. `POST /bookings/:code/capture` (no token) → 401;
  `POST /payments/paypal/webhook` (no/invalid sig) → 400; checkout for a PAYPAL
  booking maps + (with creds) returns an approve URL. Full approve→capture→webhook
  loop exercised manually with ngrok + a sandbox buyer account.

## Deps

Add `@paypal/paypal-server-sdk@^2.4.0` (direct — webpack externalizes it). Creds in
`.env` (`PAYPAL_CLIENT_ID/SECRET`, `PAYPAL_MODE=sandbox`; `PAYPAL_WEBHOOK_ID` filled
at the webhook step).
