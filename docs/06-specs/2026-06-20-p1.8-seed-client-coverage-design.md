## P1.8a вЂ” Seed (`feat/p1.8a-seed`)

Port donor `prisma/seed.ts`; adapt to our schema. **Idempotent** (re-runnable): upsert by `slug`
for destinations/tours; `(tourId, dayNumber)` for itinerary; `deleteMany`+`createMany` for departures.

- **Catalog:** TourCategories (lookup table вЂ” D-P1.5), Destinations, Tours (**EN-only** `title`/
  `summary`, no `*_vi`; **M:N** via `TourDestination` with `isPrimary`, not donor `destinationId`),
  itinerary/FAQs/policies (TourPolicy is its own model вЂ” D-P1.1), TourDepartures (seat/date guards).
- **Media:** `MediaAsset` rows (ownerType TOUR/DESTINATION, roles hero/gallery) pointing at
  Cloudinary built-in `samples/...` public_ids вЂ” so the FE renders images out of the box.
- **Users:** one CUSTOMER + one ADMIN mirrored locally (supabaseId, citext email, role). Email from
  `ADMIN_EMAILS` allowlist for the admin.
- **Self-signed PAID booking (D4):** create 1вЂ“2 bookings for the seeded customer with `status=PAID`,
  seats claimed on a real departure, a deterministic `code` (BK-вЂ¦); these unblock `POST /reviews`.
  Print the **codes** + the customer/admin identifiers.
- **Test env file:** write seeded identifiers (customer/admin ids, booking codes, a tour slug) to a
  gitignored `apps/api/.env.e2e` (or `tmp/seed-output.json`) for the e2e suite + Postman-style runs.
  **Print no secret values** (reads `apps/api/.env`).
- **Nx target:** `@tourism/api:seed` в†’ runs the seed against `DIRECT_URL` (port 5432, matches the
  migration engine via `prisma.config.ts`). Wire `.claude/commands/seed.md` (flip its "until P1 lands"
  guard). `prisma db seed` config in `package.json`/`prisma.config.ts`.

**Verification:** run the seed twice в†’ second run is a no-op-ish refresh (no duplicate rows);
`GET /tours`, `GET /destinations` return seeded data with `media[]`; printed booking code resolves
in `POST /reviews` (PAID-eligible). Gate stays green (seed is excluded from the build).

## P1.8b вЂ” shared/core API client (`feat/p1.8b-core-client`)

Generate a typed client in `@tourism/core` from the live Swagger JSON; ship a thin wrapper.

- **Deps (root devDeps):** `openapi-typescript` (generator, dev) + `openapi-fetch` (tiny runtime,
  dep of `@tourism/core`). No Java, no heavy codegen.
- **Generated types:** `libs/shared/core/src/lib/api/schema.ts` (generated; `openapi-typescript
  http://localhost:3000/api/docs-json -o вЂ¦`). Committed but marked generated.
- **Nx target:** `@tourism/core:api-types` (runs the generator; **API must be running** вЂ” the
  `/regen-types` command already documents this). Wire `.claude/commands/regen-types.md` (flip guard).
- **Client wrapper:** `createApiClient({ baseUrl, getToken })` в†’ `openapi-fetch` client with a
  middleware that (a) attaches `Authorization: Bearer <token>` when `getToken()` returns one,
  (b) unwraps our envelope so callers get `data` / typed errors instead of `{data,error,meta}`.
  Re-export typed `paths`/`components` + helper types (`ApiResponse` already in core).
- **Module boundaries:** `@tourism/core` is `scope:shared,type:data-access` вЂ” web/admin/mobile may
  import it (BLUEPRINT В§3); it imports nothing app-scoped. Generation is dev-time only.

**Verification:** `nx build @tourism/core` + typecheck green; a tiny unit test on the wrapper
(envelope unwrap + auth header injection, with a mocked fetch); `nx run @tourism/core:api-types`
regenerates deterministically (clean `git diff` on a no-op rerun).

## P1.8c вЂ” Coverage + e2e (`feat/p1.8c-coverage-e2e`)

- **Supertest happy-path e2e** (`apps/api/.../*.e2e-spec.ts` or a `test/` suite): boot the Nest app
  via `@nestjs/testing`, hit the real envelope/guards. Cover the spine:
  public catalog read в†’ (authed) booking create в†’ self-signed PAID в†’ `POST /reviews` в†’
  `GET /tours/:slug/reviews` shows it after admin approve; wishlist add/list; enquiry create + admin
  list. Auth: inject a test JWT or override `SupabaseJwtGuard` with a stub user (CUSTOMER/ADMIN).
  **DB:** runs against the seeded local DB (needs P1.8a) вЂ” documented as local/seed-gated; **not**
  added to the CI gate unless we provision a CI test DB (flagged, deferred).
- **Unit coverage в‰Ґ80%:** fill gaps вЂ” controllers without specs, any service branch under target.
  `nx test @tourism/api --coverage`; report the number; add focused specs where thin.
- Keep the existing `/gate` (lintВ·typecheckВ·testВ·build) green; e2e is a separate target
  (`@tourism/api:e2e` or `test:e2e`) so the CI gate isn't blocked on DB availability.

**Verification:** coverage report в‰Ґ80% on api logic; e2e suite passes locally against a seeded DB;
`/gate` green.

## Trust / integrity

- Seed is **dev-only**, reads `.env`, prints no secrets, targets `DIRECT_URL`. The self-signed PAID
  booking is a seed-only DB write вЂ” the runtime PAID path remains the gateway webhook + atomic CTE.
- Generated client carries no secrets; the JWT is injected at call time by the consumer via `getToken`.
